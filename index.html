The problem · the asymmetry

An attack is already in progress.

The only question is who finds out first — the attacker, or the defender. A mid-sized link carries hundreds of thousands of packets per second, hiding scans, exploits and floods.

⚔ Attacker

AUTOMATED · FAST · RELENTLESS

Needs to succeed only once. Scripts probe thousands of hosts a minute, around the clock, faster than a human can read a log.

⛨ Defender

FEW · SLOW · HUMAN

Must catch every single attack, in real time. Old tools each solve one piece — they watch, or add context, or block — but never the whole loop, safely.

The platform

One transparent platform unifies detection, response & memory.

Where commercial tools are expensive, opaque black boxes, Ma'man is one open, fully-understood system that detects, decides and responds — then enriches, explains and remembers.

⇌

NIDS — Detect

Sniffs packets, reassembles bidirectional flows, extracts 37 statistical features and classifies each flow into one of eight attack classes — or benign.

AF_PACKET capture · LightGBM gate · XGB+LGB ensemble
⛨

NIPS — Respond

When a detection is severe and confident, a playbook fires an automated response: a temporary, auto-expiring nftables block — safe by default.

Confidence-gated · TTL-bounded · reversible · audited
▤

SIEM — Remember

Every alert, block, config change and report is persisted to an embedded, ACID, full-text database and is fully searchable — the way an analyst uses Splunk or the ELK stack.

SQLite + FTS5 · hash-chained audit · CEF export
Architecture · wire to block

A packet's journey, end to end.

Capture and analysis run on background threads; the GUI never blocks. The whole journey — capture → flow → fork → response — completes in about a second.

01
CAPTURE

AF_PACKET ring buffer reads frames from the NIC, fan-out across CPUs.

02
FLOW

Frames keyed into bidirectional 5-tuple flows; 37 features per flow.

03
FORK

One fork, two detectors — ML engine and the flood detector, never overlapping.

04
DETECT

Two-stage ML classifies the class; rate logic catches volumetric floods.

05
CORRELATE

Dedup, enrich (MITRE TTP + VirusTotal), score final severity.

06
RESPOND

Playbook fires: watch-list, notify, or a real nftables block via the gate.

07
SIEM

Teed into a hash-chained audit log and the searchable event store.

The masterstroke — one fork, two detectors.

A flood and a stealthy exploit share almost nothing operationally. One detector can't do both well, so Ma'man runs two whose output spaces never overlap: zero ambiguity, zero double-counting.

Safe by default

A system that can block a host — and thought about every failure.

An auto-block on the wrong host is worse than no response at all. Every block passes four guards, on top of simulation-by-default, TTL auto-expiry, crash-safe state and an emergency kill switch.

1

Is enforcement explicitly enabled by the operator? Two flags are required to ever block live.

2

Is the address a real, valid, non-reserved IP?

3

Is it not on the never-block list — gateway, loopback, the sensor itself?

4

Is it not already blocked? An existing block just refreshes its TTL.
